Step-by-Step: When Memory Is Allocated & Freed

1. Load
Base model loaded into GPU — 3.5 GB (INT4)
Stays in memory the ENTIRE time. Never freed. Never updated.
2. Forward
Input flows through all 32 layers
Each layer: dequantize INT4→FP16 (on-the-fly), compute W×x, save output as activation
Activations accumulate: ~0.5 GB across all layers. Needed for backward pass.
3. Loss
Compare prediction to correct answer
Scalar value — negligible memory.
4. Backward
Gradient flows backward through all 32 layers
Each layer: use saved activation to compute gradient (0.07 GB)
Activation for that layer is then DISCARDED (freed from memory)
STE: gradient flows to LoRA params only, INT4 weights get no gradient.
5. Update
Adam updates LoRA weights
m (momentum) + v (velocity) updated using gradient (0.3 GB total)
LoRA weights (B and A) updated: W_new = W_old − lr × m/√v
Adam states persist across steps. Base model weights NEVER change.
6. Repeat
Next batch starts from Step 2
Old activations are gone. New forward pass generates new ones.
Gradients are overwritten each step. Adam states accumulate.

🔑 The key memory-saving trick in QLoRA:
Base model (3.5 GB) stays in INT4 — no Adam, no gradients, no updates. That's 96% of the model that costs NOTHING beyond storage.
Gradients (0.07 GB) exist only for the 37M LoRA params, not the 7B frozen ones.
Adam (0.3 GB) only tracks m+v for LoRA params.
Activations (0.5 GB) are needed regardless — but they're temporary (freed after backward).
