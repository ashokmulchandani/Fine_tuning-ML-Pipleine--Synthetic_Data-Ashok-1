Adapter Merging — Why "Merge, Never Stack"

Multi-stage fine-tuning: Domain → Instruction → Preference. Each stage must bake into the model.

❌ Stacking (Wrong Way) — Each Adapter Fixes the Previous One

Step 1: Base Model
W₀ = −2.30
Trained on internet text
Step 2: Domain FT

ΔW₁ = +0.15

Effective: −2.30+0.15 = −2.15
Step 3: Instruction FT
❌ STACKED!

ΔW₁
+
ΔW₂ = −0.30
2 adapters stacked
Effective: −2.30+0.15−0.30 = −2.45
Step 4: DPO
❌ TRIPLE STACKED!
→
W₀
+
ΔW₁
+
ΔW₂
+
ΔW₃ = +0.60
3 adapters stacked
Effective: −2.30+0.15−0.30+0.60 = −1.85
🔴 What went wrong? ΔW₂ was trained while ΔW₁ was still active. It learned to fix errors created BY ΔW₁ — not errors in the base model. ΔW₃ learned to fix errors from BOTH previous adapters. Each new adapter is patching patches of patches. The weight drifts from −2.30 to −1.85 — completely lost its original meaning.

This is "catastrophic drift." Each adapter was trained in a world where the previous adapter was still active. After 3 stages, they're all fighting each other.

✅ Merging (Right Way) — Bake In Before Next Stage

Step 1: Base Model
W₀ = −2.30
Step 2: Domain FT
→
W₀
+
ΔW₁
Train domain adapter
⚡ MERGE!
→
W₁ = W₀ + ΔW₁ = −2.15
Adapter baked in!
Adapter 1 is GONE. Just one model now.
Step 3: Instruction FT
→
W₁
+
ΔW₂ (fresh)
Train instruction adapter from scratch on MERGED model
⚡ MERGE!
→
W₂ = W₁ + ΔW₂ = −2.45
Clean base for next stage
Step 4: DPO
→
W₂
+
ΔW₃ (fresh)
Train from the merged model — clean start
🟢 Why this works: After each MERGE, the adapter vanishes and becomes part of the model. The next adapter sees a CLEAN model with no active adapters. ΔW₂ learns from W₁ (the improved model), not from W₀+ΔW₁ (model with an active patch). Each stage builds on the PREVIOUS MERGED result, not on a stack of still-active patches.

📊 Stacking vs Merging — The Same Weight Through 3 Stages

❌ Stacking (Don't Merge)

Start: W₀ = −2.30
+Domain: W₀ + A₁ = −2.15 (A₁ trained on W₀)
+Instr: W₀ + A₁ + A₂ = −2.45 (A₂ trained on W₀+A₁)
+DPO: W₀ + A₁ + A₂ + A₃ = −1.85 (A₃ trained on W₀+A₁+A₂) ← DRIFT!

✅ Merging (Correct Way)

Start: W₀ = −2.30
+Domain: W₁ = −2.15 (A₁ merged into W₀)
+Instr: W₂ = −2.45 (A₂ trained on W₁, then merged)
+DPO: W₃ = −2.30 (A₃ trained on W₂, then merged) ← STABLE

🎨 The Paint Analogy

❌ Stacking = Wet Paint on Wet Paint

You apply red paint (Domain). It's still wet. You apply blue paint (Instruction) ON TOP of wet red = MUDDY PURPLE. You apply yellow paint (DPO) ON TOP of wet purple = MUDDY BROWN. Final result: nothing like what you wanted.

✅ Merging = Dry Between Coats

You apply red paint (Domain). Let it DRY (merge). You apply blue paint (Instruction) on the dry red = CLEAN BLUE ON RED. Let it DRY (merge). You apply yellow paint (DPO) on dry surface = CLEAN YELLOW. Final result: each layer is pure. No muddy mixing.

💻 How to Merge in Code

# Load base model + trained LoRA adapter model = PeftModel.from_pretrained(base_model, "domain_adapter") # MERGE: bake adapter into base weights, then delete adapter model = model.merge_and_unload() # ↑ Now model is a plain transformers model. No LoRA. Clean slate. # Apply fresh LoRA for next stage model = get_peft_model(model, lora_config_instruction) model.train() # Trains NEW adapter from scratch on merged weights model.save_pretrained("instruction_adapter") # Merge again, then DPO model = model.merge_and_unload() model = get_peft_model(model, lora_config_dpo)
🧠 The Golden Rule:
merge_and_unload()
before every new training stage.
No exceptions. No shortcuts. Merge, then proceed.
